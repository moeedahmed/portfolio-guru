# Portfolio Guru Core Edge - 2026-07-06

Source: Moeed critique in Founder Content topic, 2026-07-06.
Purpose: product positioning and spec input. Public copy must still distinguish current live capability from planned/soon capability.

## Core Positioning

Portfolio Guru is not primarily a writing assistant. Doctors already use ChatGPT and similar tools for drafting.

The edge is end-to-end portfolio filing: capture evidence where the doctor already is, use AI to reduce selection and drafting load, then file the result into the actual portfolio workflow as a Kaizen draft with human approval.

## User Surface

- Current primary surface: Telegram.
- Planned convenience surface: WhatsApp.
- The product should live where the doctor naturally captures evidence during or after a shift, not require a separate admin session first.

## Cognitive Load To Remove

- File a case from the bedside.
- Send a picture of notes.
- Send a voice reflection.
- Send clinical media or files that should become portfolio attachments, subject to privacy and product safety gates.
- Use all of the above as drafting context.
- Keep drafts simple and reviewable.
- Match the doctor's own voice profile when one has been built from previous filings.
- Suggest or select the appropriate Kaizen ticket/form type.
- Create multiple tickets from one clinical run when appropriate.
- Let the doctor improve or add information before filing.
- Save the output into Kaizen as a draft after approval, leaving the doctor to review later inside the portfolio.

## Safety Boundary

The product files drafts only. It must not submit, approve, sign, send to supervisors, or imply official ARCP outcome assurance.

## Content Pipeline Rule

Portfolio Guru content should not flatten the product into generic ARCP visibility or AI drafting. It should preserve the actual wedge: AI-assisted capture, selection, drafting, attachment handling, and approval-gated filing into Kaizen.

When a Portfolio Guru approval draft is reviewed, the discussion should happen in the Portfolio/product topic so critique can update both content positioning and product implementation context.
